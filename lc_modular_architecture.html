LC APP MODULAR ARCHITECTURE

Independently buildable domains for request, permit, execution, return, audit, and notifications

Architecture goal

Why modular for LC

🧩

Independent delivery tracks

Teams can build request, approval, field execution, and substation return flows in parallel with stable APIs and shared identity contracts.

🛡️

Safety and audit isolation

Safety-critical actions such as permit issue, LC return, override, and revocation stay in dedicated services with immutable logs and stronger authorization.

📶

Offline-first field support

Field modules can cache work orders, crew checklists, and return drafts independently without blocking approval or dashboard modules.

🔁

Clear integration boundaries

Notification, document, reporting, and master-data modules subscribe to domain events instead of being tightly embedded in workflow code.

Principles

Design principles

Domain-led modules

Separate business capabilities, not technical layers, so each module maps to a real operational owner.

Contract-first APIs

REST and event contracts versioned from day one to avoid cross-team regressions.

Shared auth core

Identity, roles, sections, and duty rosters come from one trusted authorization service.

Immutable audit trail

Every approve, issue, avail, pause, revoke, and return event is append-only and timestamped.

Offline sync boundaries

Only the field app syncs delayed commands; approval and substation actions stay online and synchronous.

Config over code

Approval matrix, hierarchy, station mapping, and form rules stay configurable for each utility zone.

Integration contracts

Core APIs and events

Synchronous APIs

Create request, submit approval decision, issue permit, avail permit, return permit, fetch dashboard queues, and download permit document.

Domain events

LC_REQUESTED, LC_APPROVED_L1, LC_APPROVED_L2, LC_ISSUED, LC_AVAILED, WORK_COMPLETED, LC_RETURNED, LC_CLOSED, LC_REVOKED.

Contract rules

Each event carries permitId, actorId, role, stationId, feederId, timestamp, and correlationId for traceability.

Consumer modules

Notifications, audit, reporting, and document generation subscribe without changing core workflow services.

Data ownership

Data domains and boundaries

Workflow DB

LC requests, approval states, role assignments, SLA deadlines, and current permit status.

Permit DB

Issue checklist, operator actions, field avail details, work completion, return and closure artifacts.

Master DB

Feeders, substations, sections, employee registry, hierarchy, and standard work templates.

Audit store

Append-only event log with device, location, IP, and pre/post payload snapshots.

Object storage

Photos, signatures, PDF permits, risk sheets, and closure evidence.

Analytics mart

Read-optimized datasets for dashboards, performance reporting, and safety trend analysis.

Security

Security and governance controls

Role and duty checks

Only mapped AE can raise, AEE and EE can approve, line staff can avail, and substation operators can issue or return based on station duty roster.

Safety state machine

Invalid transitions are blocked, for example execution cannot start before issue, and closure cannot happen before return verification.

Strong evidence capture

Geo-stamped photos, signatures, timestamps, and optional second-factor confirmation for permit issue and closure.

Operational governance

Configurable maker-checker rules, zone-level reporting, audit export, retention, and incident replay capabilities.

Recommended rollout

PHASE BY DOMAIN, NOT BY SCREEN

Phase 1

Identity, master data, LC request, approval engine, and basic notifications.

Phase 2

Permit issue, field execution, offline sync, evidence, and return workflow.

Phase 3

Documents, analytics, audit replay, advanced escalation, and enterprise integrations.
